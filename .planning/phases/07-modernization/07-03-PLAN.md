<task type="auto">
  <name>Task 1: Add PSR-4 autoload mapping and regenerate autoloader</name>
  <files>composer.json</files>
  <action>
In `composer.json`, replace the existing `"autoload"` section:

```json
"autoload": {
    "classmap": ["comment-popularity.php"]
}
```

With a PSR-4 mapping:

```json
"autoload": {
    "psr-4": {
        "CommentPopularity\\": "inc/"
    },
    "exclude-from-classmap": ["inc/lib/"]
}
```

The `exclude-from-classmap` prevents Composer from scanning the vendor directory (which is `inc/lib/` due to the non-standard `vendor-dir` setting).

Note: The existing class file naming convention (`class-comment-popularity.php`, `class-visitor.php`) does NOT follow PSR-4's `ClassName.php` convention. However, Composer's PSR-4 autoloader uses the class name to derive the file path, so we need a `classmap` fallback for these files. Use BOTH psr-4 and classmap:

```json
"autoload": {
    "psr-4": {
        "CommentPopularity\\": "inc/"
    },
    "classmap": [
        "inc/class-comment-popularity.php",
        "inc/class-visitor.php",
        "inc/widgets/class-widget-most-voted.php",
        "inc/widgets/experts/class-widget-experts.php",
        "admin/class-comment-popularity-admin.php"
    ],
    "exclude-from-classmap": ["inc/lib/"]
}
```

This lets Composer find the classes by scanning (classmap) while the PSR-4 entry enables future classes with standard naming to autoload without classmap entries.

After editing, run `composer dump-autoload` to regenerate the autoloader.

Verify with: `php -r "require 'inc/lib/autoload.php'; echo class_exists('CommentPopularity\HMN_Comment_Popularity') ? 'OK' : 'FAIL';"` — should print `OK`.
  </action>
